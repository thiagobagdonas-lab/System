## 1. Contexto e Objetivo

A Setter está implementando um servidor MCP (Model Context Protocol) próprio como parte da adoção estratégica da plataforma Claude (Anthropic) para operações de M&A. O MCP é o protocolo padrão que permite ao Claude acessar dados internos, executar automações e consultar bancos de dados em tempo real durante as análises dos analistas.

**Objetivo desta reunião:** Alinhar a divisão de responsabilidades entre TI e a área de IA/Dados para o provisionamento, segurança e manutenção contínua da infraestrutura de VPS que hospedará o servidor.

**Resultado esperado:** TI entrega uma VPS hardened e assume a manutenção de segurança da infraestrutura. A área de IA/Dados assume o deploy e manutenção da aplicação, com posse exclusiva das chaves de API.

---

## 2. O Que é o MCP Server (contexto técnico)

O MCP Server é um processo Python (FastMCP + Uvicorn) que roda na VPS e expõe endpoints HTTP autenticados. Os clientes são os aplicativos Claude (Claude Desktop, Claude Code, Claude Cowork) instalados nas máquinas dos usuários da Setter.

```
[Claude — máquinas dos analistas]
        │
        │  HTTPS + Bearer Token
        ▼
[nginx — reverse proxy + TLS termination]
        │
        │  HTTP interno (127.0.0.1:8000)
        ▼
[Uvicorn → FastMCP Application]
        │
        ├── SQLite (banco interno de deals)
        ├── Scripts Python/Shell (automações ETL)
        └── Audit log (append-only, imutável)
```

**O que o servidor expõe:**
- **Tools:** funções que o Claude pode invocar — inserir deals, disparar ETL, gerar relatórios
- **Resources:** consultas read-only ao banco — pipeline de deals, valuations, research setorial
- **Health endpoint:** `GET /health` — sem autenticação, para monitoramento

**O que o servidor NÃO faz:**
- Não acessa a internet (apenas recebe conexões de entrada)
- Não armazena dados em memória entre sessões (stateless por request)
- Não executa código arbitrário (allowlist explícita de scripts autorizados)

## 3. Divisão de Responsabilidades

Esta é a separação proposta entre TI e IA/Dados:

### 3.1 Responsabilidade do TI — Infraestrutura e Segurança

| Domínio | Tarefa | Periodicidade |
|---|---|---|
| **Provisionamento** | Criação da VPS com Ubuntu 22.04 LTS | Uma vez |
| **Acesso inicial** | Criação do usuário `setter` com sudo, chave SSH | Uma vez |
| **SSH hardening** | Desabilitar root login, desabilitar password auth | Uma vez |
| **Firewall (UFW)** | Liberar 22/80/443, bloquear todo o resto | Uma vez |
| **Fail2ban** | Configurar banimento por tentativas SSH e nginx | Uma vez |
| **Certbot / TLS** | Emitir e renovar certificado Let's Encrypt para `mcp.setter.net.br` | Automático (renovação 90 dias) |
| **OS updates** | `unattended-upgrades` para patches de segurança | Automático (diário) |
| **Monitoramento** | Alertas de CPU/memória/disco acima de 80% | Contínuo |
| **Acesso de emergência** | Console de recuperação via provedor (acesso out-of-band) | Sob demanda |
| **Auditoria de acesso** | Revisão mensal de `/var/log/auth.log` e eventos fail2ban | Mensal |

### 3.2 Responsabilidade da Área de IA/Dados — Aplicação

| Domínio | Tarefa |
|---|---|
| **Deploy** | Código da aplicação, virtualenv, dependências Python |
| **Configuração da aplicação** | Arquivo `.env` com as API keys (posse exclusiva) |
| **Banco de dados** | Schema SQLite, backups diários, rotação de dados |
| **systemd service** | Arquivo de serviço, start/stop/restart da aplicação |
| **nginx** | Configuração do `server block` para o MCP |
| **Manutenção do código** | Updates de versão do FastMCP, novas tools, novos scripts |
| **API keys** | Geração, rotação e distribuição das chaves de autenticação |
| **Audit log** | Monitoramento e análise de eventos de auditoria da aplicação |

### 3.3 Fronteira crítica — o que cada parte NUNCA toca

> **TI não acessa:**
> - O arquivo `config/.env` (contém as API keys)
> - O conteúdo do banco SQLite (`data/setter.db`)
> - Os audit logs da aplicação (`logs/audit.log`)
>
> **IA/Dados não assume:**
> - Manutenção do kernel e pacotes do OS
> - Regras de firewall e fail2ban
> - Renovação de certificados TLS
> - Acesso de emergência via console do provedor

Esta separação respeita o princípio de menor privilégio e garante que nenhuma das partes precisa da outra para executar suas funções rotineiras.

## 4. Especificações Técnicas da VPS

### 4.1 Requisitos mínimos

| Recurso | Mínimo | Recomendado |
|---|---|---|
| vCPU | 1 | 2 |
| RAM | 2 GB | 4 GB |
| Disco SSD | 20 GB | 40 GB |
| Rede | 100 Mbps | 200 Mbps |
| OS | Ubuntu 22.04 LTS | Ubuntu 22.04 LTS |
| IP | 1 IPv4 estático | 1 IPv4 estático |

**Estimativa de custo:** R$ 80–150/mês (Hetzner CX21 ou DigitalOcean equivalente). Custo aumenta apenas se SQLite ultrapassar ~50 GB — improvável no horizonte de 12 meses.

### 4.2 Provedor recomendado

- **Hetzner Cloud** (Helsinki ou Ashburn): melhor custo-benefício, ~€4/mês para CX21
- **DigitalOcean** (São Paulo): latência menor para o Brasil, ~$24/mês para Droplet equivalente

Ambos fornecem console out-of-band, snapshots automatizados e IPv4 estático incluso.

### 4.3 Configuração de rede necessária

- **DNS:** Registro A apontando `mcp.setter.net.br` → IP da VPS (TI ou responsável pelo DNS da Setter)
- **Portas abertas no firewall externo:** 22 (SSH), 80 (HTTP→redirect), 443 (HTTPS)
- **Nenhuma outra porta deve ser exposta** — o processo Python escuta exclusivamente em `127.0.0.1:8000`

## 5. Modelo de Segurança

### 5.1 Camadas de defesa

```
Internet
   │
   ├─ [UFW] Bloqueia tudo exceto 22/80/443
   │
   ├─ [Fail2ban] Bane IPs após 3 falhas SSH / 5 falhas HTTP em 10 min
   │
   ├─ [nginx] TLS 1.2+, HSTS, rate limiting (10 req/s), sem buffering
   │
   ├─ [FastMCP + StaticTokenVerifier] Bearer token obrigatório por tier
   │       ├─ Tier read  → analistas (consultas)
   │       ├─ Tier write → automações internas
   │       └─ Tier admin → responsável IA/Dados
   │
   ├─ [SQLite WAL + queries parametrizadas] Zero SQL injection possível
   │
   └─ [Audit log append-only] chattr +a — nem root deleta
```

### 5.2 Autenticação — o que TI precisa saber

O servidor usa Bearer tokens (API keys) gerados e gerenciados exclusivamente pela área de IA/Dados. O TI **não precisa e não deve** conhecer os valores dessas chaves. A autenticação é tratada na camada de aplicação (FastMCP), não no nginx.

O nginx valida apenas TLS e aplica rate limiting. Não faz autenticação de conteúdo.

### 5.3 Vetor de ataque mapeado — prompt injection via MCP

O principal risco específico desta aplicação é **prompt injection**: um documento malicioso aberto por um analista poderia, em tese, tentar instruir o Claude a executar queries não autorizadas via MCP. As defesas implementadas:

- Queries SQL sempre parametrizadas (sem interpolação de string)
- Allowlist explícita de scripts que podem ser invocados
- Parâmetros passados via stdin JSON, nunca como args de shell
- Audit log de toda operação, incluindo parâmetros recebidos
- Tools de escrita requerem campo `justificativa` obrigatório

## 7. Manutenção Contínua — O que o TI Mantém

### Rotina automática (sem intervenção manual esperada)

- **Patches de segurança:** `unattended-upgrades` aplica automaticamente patches do kernel e pacotes críticos
- **Renovação TLS:** Certbot renova o certificado a cada 60–90 dias via `cron` — falha gera alerta
- **Fail2ban:** opera de forma autônoma, bane e desbane IPs conforme regras configuradas

### Rotina mensal (intervenção ativa do TI)

- Revisão de `/var/log/auth.log` — identificar tentativas de acesso suspeitas
- `sudo fail2ban-client status` — revisar IPs banidos recorrentes
- `df -h` — verificar crescimento de disco (alertar IA/Dados se > 70%)
- `sudo apt list --upgradable` — verificar se há pacotes pendentes fora do escopo do `unattended-upgrades`

### Atualizações de kernel com reboot

Quando o kernel for atualizado (raro, mas necessário), o processo é:

1. TI notifica IA/Dados com 24h de antecedência
2. IA/Dados confirma janela de manutenção (evitar horário comercial)
3. TI executa `sudo reboot`
4. systemd reinicia automaticamente o serviço MCP após o boot
5. IA/Dados confirma que `GET https://mcp.setter.net.br/health` retorna `{"status":"ok"}`

## 8. Modelo de Comunicação e Escalação

| Situação | Responsável primário | Ação |
|---|---|---|
| VPS inacessível via SSH | TI | Acesso via console out-of-band, investigar SO |
| Serviço MCP fora do ar (nginx OK, SSH OK) | IA/Dados | `sudo systemctl restart setter-mcp` |
| Certificado TLS expirado | TI | `sudo certbot renew` |
| Tentativas de acesso suspeitas (auth log) | TI | Revisar fail2ban, reforçar regras UFW se necessário |
| Disco acima de 80% | TI alerta → IA/Dados resolve | IA/Dados limpa backups antigos ou expande banco |
| Rotação de API keys | IA/Dados | Atualiza `.env`, reinicia serviço — TI não é envolvido |
| Vulnerabilidade crítica (CVE) no OS | TI | Patch e reboot em janela de manutenção acordada |
| Nova automação ou tool no servidor | IA/Dados | Deploy via git pull, restart do serviço — TI não é envolvido |
