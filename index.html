Como adotar o Claude com segurança.

Este briefing consolida os três ambientes de execução — Chat, Cowork e Code — e como Projects + Skills se encaixam para padronizar o time. Onde o código executa, qual o nível de isolamento, e como distribuir scripts para analistas sem expor a operação.

Pergunta central
Qual ambiente usar para cada caso, sabendo que o código sairá da minha máquina e rodará nos analistas?
Recomendação
Claude Code com sandbox para desenvolvimento, Code na Web para execução crítica em analistas (VM descartável + repo auditado = trilha defensável), Projects + Skills para padronizar o time.
Risco crítico hoje
Cowork não gera audit log em nenhum plano — incluindo Enterprise. Não use para fluxos regulados.
Baseado na documentação oficial Anthropic, GitHub anthropics/claude-code e PromptArmor Pular para o sumário executivo
01

Os quatro produtos · panorama em 30 segundos

Mesmo modelo Claude por trás. O que muda é onde o agente executa e o que ele acessa. Detalhes aprofundados nas próximas seções.

I

Claude Chat

Web · Mobile · Desktop

Conversa de uso geral. Executa código apenas no sandbox da Anthropic (artifacts).

  • Para: qualquer usuário
  • Execução: infra Anthropic
  • Acesso local: nenhum
II

Claude Cowork

App desktop · híbrido

Voltado a analistas. Roda em VM Linux local + loop de execução também no host (fora da VM) para arquivos e MCP.

  • Para: áreas de negócio
  • Execução: VM local persistente
  • Acesso local: pasta conectada
III

Claude Code

CLI · Desktop · Web

Para devs. Acesso direto ao OS; isolamento opt-in via sandbox/Docker; VM gerenciada na versão Web.

  • Para: desenvolvedores
  • Execução: OS direto ou VM
  • Acesso local: total (sem sandbox)
IV

Projects

Funcionalidade transversal

Espaço persistente com instruções, arquivos, memória e histórico. Não é ambiente de execução — é onde o contexto se acumula.

  • Para: times recorrentes
  • Persistência: arquivos + memória
  • Compartilha: Team/Enterprise
02

Conceitos fundamentais · clique para expandir

Os blocos que se combinam em uma adoção corporativa. Skills, Projects, Sub-agents, Connectors, Plugins, MCP.

SKILL "Como fazer" — procedimento reutilizável

Arquivo SKILL.md com instruções, conhecimento de domínio e gatilhos que ensinam Claude a executar um workflow específico de forma repetível. Funciona como um POP corporativo.

  • Carregam contexto apenas quando relevante (evita "context rot")
  • Movem o Claude de "sugestivo" para "determinístico"
  • Globais — não ficam presas a um projeto
  • Em março/2026 viraram padrão aberto: agentskills.io — Atlassian, Figma, Stripe, Notion, Zapier
  • Incluídas em Max, Pro, Team e Enterprise sem custo adicional
PROJECT "Onde trabalhar" — espaço persistente do time

Espaço de trabalho em torno de um assunto: documentos, instruções customizadas (system prompt), memória e histórico de conversas — tudo no mesmo lugar.

  • Contexto sempre disponível — analista não parte do zero
  • Arquivos de referência: brand book, glossário, exemplos passados
  • Team/Enterprise: compartilhamento e controle de permissões (usar × editar)
  • Enterprise: isolamento entre projetos — útil para separar clientes/mesas
PLUGIN Skill empacotada com Connectors e Sub-agents

Um Plugin agrupa três elementos:

  • Skills — instruções e conhecimento de domínio
  • Connectors — acesso governado a dados externos (FactSet, Google Drive, Slack…)
  • Sub-agents — instâncias Claude adicionais para subtarefas

A skill (instruções) funciona igual em Chat, Cowork e Code. Já sub-agents e hooks têm restrições — só funcionam plenamente no Claude Code (matriz na Seção 04).

SUB-AGENT Instância Claude separada para uma subtarefa

Instância Claude isolada com contexto próprio que recebe uma tarefa, executa e retorna apenas o resultado final ao agente principal.

  • Uso ideal: tarefas independentes que não precisam se comunicar
  • Vantagem: protege o contexto principal; exploração fica na "bolha"
  • Custo: workflows multi-agente consomem 4–7× mais tokens que sessão de agente único
  • A partir do Claude Code v2.1.172 (10/jun/2026): aninhamento até 5 níveis
CONNECTOR · MCP Como Claude acessa dados externos com governança

Connectors são integrações governadas: FactSet, Google Drive, Slack, S&P, Daloopa, Moody's. Implementados via MCP (Model Context Protocol).

MCP Gateway é o padrão corporativo: um proxy central que detém as credenciais. O agente do analista chama o gateway sem nunca ver a chave da API — auditável e revogável.

Nunca colocar API keys ou credenciais raw em arquivos do projeto que serão distribuídos.

DYNAMIC WORKFLOWS Centenas de sub-agents paralelos por sessão (Opus 4.8, mai/2026)

Permite que Claude planeje uma tarefa, dispare múltiplos sub-agents paralelos e verifique resultados antes de entregar.

  • Limite atual: 16 agentes concorrentes, 1.000 agentes por execução total
  • Disponível em Claude Code (research preview) — planos Enterprise, Team e Max
  • Cowork: ainda não disponível

A única seção que você precisa para entender risco de execução. Clique nas abas para o detalhamento por ambiente.

Detalhamento por ambiente

Onde roda

  • Sandbox isolado gerenciado pela Anthropic (tecnologia gVisor da Google para isolamento kernel-level)
  • Apenas ferramentas restritas: artifacts (JavaScript em iframe) e analysis tool (Python)
  • Sem acesso ao sistema operacional do usuário

Orquestração de agentes

  • Não existe orquestração real no Chat
  • Skills funcionam apenas como instruções
  • Sub-agents nativos via plugin ficam desabilitados na interface

"É 100% seguro rodar código desconhecido?"

  • Sua máquina está protegida — código não escapa do sandbox
  • Mas não é absolutamente seguro: sandboxes têm escapes documentados (CVEs raros mas reais)
  • O script pode ler tudo que está naquela conversa — arquivos enviados, dados colados, contexto
  • Pode tentar chamadas de rede (limitadas mas existentes em artifacts)
  • Output pode conter prompt injection que manipula o resto da conversa

Regra prática

  • Para código desconhecido: usar sessão limpa, sem upload de arquivos sensíveis
  • Não colar dados confidenciais antes nem depois
  • Validar o output manualmente — não confiar no resultado cegamente
  • Audit log: parcial (não defensável para fluxo regulado)

Dois ambientes simultâneos

  • Dentro da VM (isolado): bash, pip/npm install, scripts, compilação
  • Ubuntu 22.04 via Apple Virtualization Framework (macOS) ou Hyper-V (Windows)
  • Fora da VM (sem sandbox): leitura/escrita nas pastas conectadas via VirtioFS
  • Web fetches, MCP servers locais e Computer Use também rodam no host

VirtioFS — como o arquivo "sai" da VM

VM Linux (Ubuntu 22.04)
└── /mnt/.virtiofs-root/shared/pasta-do-usuario/
        ↕ VirtioFS (bidirecional)
Host (macOS / Windows)
└── ~/pasta-do-usuario/ ← arquivo aparece aqui
  • Output do script aparece instantaneamente na pasta do host
  • Não há etapa de cópia ou transferência manual

O que código malicioso pode (e não pode) fazer

Ler/escrever na pasta conectada✓ Sim
Enviar dados para servidor externo✓ (sujeito a egress)
Acessar SSH keys, ~/.aws, ~/.ssh✕ Se fora da pasta
Acessar outros diretórios✕ Bloqueado pela VM
Persistir entre sessões✓ VM não é descartada

Vulnerabilidades estruturais

  • Sem audit log em nenhum plano (incluindo Enterprise)
  • EDR / DLP / SIEM corporativos não enxergam a VM
  • Computer Use roda no host — sem sandbox algum

Onde roda

Direto no OS, com as permissões do usuário. Sem isolamento por padrão.

Camadas de controle

  • Permissões Allow/Ask/Deny por ferramenta — contornável via shell
  • Sandbox kernel-level (opt-in)
  • Hooks PreToolUse — inspeciona e bloqueia operações
  • Audit log disponível no plano Enterprise

VM Anthropic descartável

  • VM isolada gerenciada pela Anthropic
  • Descartada ao encerrar a sessão — sem resíduo na máquina

Pré-requisitos

  • Repositório no GitHub conectado
  • Código precisa estar commitado — não aceita scripts colados

Pré-instalado na VM

  • Linguagens: Python 3.x, Node.js 20/21/22, Ruby, PHP, Java, Go, Rust, C/C++
  • Bancos: PostgreSQL 16, Redis 7.0
  • Infra: Docker, git, utilitários comuns
  • Acesso padrão a PyPI, npm, RubyGems, crates.io

Limites importantes

  • Timeout bash: 2 minutos (configurável via BASH_MAX_TIMEOUT_MS)
  • Janela de uso: 5 horas de uso ativo a cada 5 horas (cota rolante), compartilhada com Chat e Cowork no mesmo plano
  • Não adequado para processos contínuos de longa duração

Quando usar

  • Scripts críticos que acessam dados sensíveis
  • Fluxos que precisam de rastreabilidade
  • VM descartável + repo auditado = trilha defensável

  4. Padrão: herda o modelo da sessão principal (inherit)

BUG · GitHub Issue #47488 · ABERTO

No Cowork, todo sub-agent é silenciosamente forçado para Haiku

Causa raiz

  • O Electron host do Claude Desktop injeta CLAUDE_CODE_SUBAGENT_MODEL=claude-haiku-4-5-20251001 em todo subprocess CLI
  • A variável de ambiente sobrescreve qualquer escolha do usuário
  • Campo defaultSubagentModel existe no schema mas nunca recebe valor

Afeta

  • Parâmetro model na tool Agent/Task
  • Campo model: no frontmatter dos agentes
  • Estado do comando /model

Impacto prático

  • Skills desenhadas com Opus + Sonnet rodam tudo em Haiku
  • Resultado visivelmente inferior em raciocínio complexo
  • Sem nenhum aviso ao usuário

Status

  • Presente desde o lançamento (março/2026)
  • Sem assignee, sem milestone, sem PR de fix visível
  • Confirmado em Windows, Max 5× e Max 20×
  • Fix é pequeno: falta apenas um writer para defaultSubagentModel

05

Segurança · os três vetores que importam

Onde o código executa muda o blast radius. Prompt injection, supply chain e visibilidade zero são os riscos sistêmicos.

1

Prompt Injection

  • Atacante embute instruções em documento, página web ou resposta de API
  • O agente lê e executa como se fosse comando legítimo
  • Difícil de resolver — o agente precisa processar conteúdo não-confiável
Caso real · jan/2026 · PromptArmor
  • Documento Word com texto branco invisível (1 ponto)
  • Cowork fez upload de documentos financeiros para conta do atacante
  • Sem exploit, sem malware — apenas abrir o arquivo
Vetores em multi-agent
  • Documentos processados por sub-agents
  • Respostas de APIs via MCP tools
  • Registros de banco retornados por SQL
  • READMEs ou comentários em código lido
2

Supply Chain

  • Claude instala dependências de PyPI/npm/RubyGems sem auditar versões
  • Apenas hashes padrão dos gerenciadores de pacote
  • Pacote comprometido vira Remote Access Trojan no momento da execução
Eventos 2026
  • Mar/2026: axios (npm) comprometido com RAT, mesmo mês de incidente com Claude Code
  • Q1/2026: CVEs originados de código gerado por IA cresceram de 6 para 35/mês
  • Documentação de API envenenada causando instalação automática de deps maliciosas
Mitigação
  • Versões pinadas no requirements.txt / package-lock.json
  • Code review humano nas dependências novas
  • Distribuir apenas após auditoria
3

Visibilidade Zero

  • Cowork não entra em Audit Logs, Compliance API nem Data Exports
  • Em nenhum plano — incluindo Enterprise
  • EDR / DLP / SIEM corporativos não enxergam a VM por design
Consequência operacional
  • Sem log do que foi executado, lido ou enviado
  • Ponto cego para compliance de fundo regulado
  • Investigação de incidente fica sem trilha
Mitigação
  • Claude Code Enterprise — audit log ativo
  • Claude Managed Agents — logs nativos no Console

08

MCP Server próprio · vantagens e armadilhas

Esse "MCP Server próprio" é o que cumpre na prática o papel de "MCP Gateway" descrito na Seção 02 — o proxy corporativo que centraliza credenciais, audit log e regras de negócio. Ter seu próprio vira o eixo de governança entre Claude e os sistemas da gestora — mas também um ponto único de falha se mal configurado. Esta seção responde: quando vale, como hospedar com segurança, e quando dá para fazer mais simples.

Vantagens de ter um MCP Server próprio

01

Banco de dados consolidado

Único caminho para consolidar o trabalho de todos os analistas em uma base estruturada — análises, métricas, histórico de decisões. Sem MCP próprio, cada analista trabalha em silo isolado, sem reaproveitamento.

02

Credenciais centralizadas

Todas as API keys (FactSet, S&P, Daloopa, banco interno, Bloomberg) ficam no servidor. Analista nunca vê nem precisa armazenar.

03

Audit log próprio

Toda chamada de connector registrada com usuário, timestamp e parâmetros. Suplementa o audit log do Code Enterprise.

04

Mascaramento de dados

Camada de transformação antes do conteúdo chegar ao Claude — remove CPFs, PIIs, números de contas, posições internas.

05

Revogação por usuário

Demitir analista = revogar token no MCP, sem ter que trocar as API keys upstream caras (FactSet, S&P).

06

Rate limit e cache

Controla consumo por usuário ou projeto. Cacheia respostas para reduzir custo das APIs externas pagas.

07

Agregação de fontes

Une múltiplos sistemas internos (Bloomberg, Oracle, BI, planilhas) em um único endpoint padronizado para o Claude consumir.

08

Lógica de negócio centralizada

Regras proprietárias da gestora (mapeamento de tickers, cálculos internos, normalizações) ficam no MCP. Mudanças não exigem atualizar scripts em cada analista.

Vetores de ataque · cenário mal configurado

Os seis vetores abaixo se manifestam apenas no contra-exemplo: VPS deixada na internet sem firewall e sem autenticação forte. Com o MCP configurado no Modo 1 (controle por firewall) ou Modo 2 (controle por autenticação), descritos mais abaixo, a maioria deixa de ser vetor real — ver subseção seguinte.

Endpoint exposto

VPS pública com IP roteável e portas abertas para qualquer um. Atacante encontra via Shodan ou scanning automatizado em horas.

Comprometimento total

Se o atacante invade o servidor, ele tem todas as credenciais corporativas dentro dele. Bloomberg, FactSet, banco interno, tudo de uma vez.

Exfiltração disfarçada

MCP Server tem permissão para chamar APIs externas. Atacante usa essa permissão para enviar dados internos a servidores que controla.

Dependências envenenadas

Uma única biblioteca comprometida no projeto do MCP Server (npm, pip) basta para vazar credenciais em runtime.

Logs expostos

Logs do servidor contêm parâmetros de chamadas — incluindo dados sensíveis que passaram pelo proxy. Se a VPS é invadida, logs vazam junto.

Backdoor persistente

VPS pública é difícil de monitorar sem ferramenta dedicada. Atacante pode manter acesso por meses sem ser detectado.

Risco real quando o MCP está bem configurado

Quando o MCP está no Modo 1 (controle por firewall na camada de rede) ou Modo 2 (controle por autenticação na camada de aplicação) — ambos descritos mais abaixo — a superfície de ataque cai drasticamente. O que sobra são riscos comuns a qualquer serviço corporativo, mitigáveis com processo:

Baixa probabilidade

Descoberta via Certificate Transparency

Todo certificado TLS emitido vira público em logs de CT (crt.sh). Subdomínio fica descobrível. Não é vetor real se a autenticação está correta — atacante precisa furar o auth.

Média probabilidade

Vulnerabilidade no código do MCP

Bug no servidor pode vazar dados mesmo com auth válida — SQL injection, path traversal, parsing inseguro. Mitigação: code review, fuzzing, libs maduras.

Média probabilidade

Supply chain de dependências

Pacote npm/pip comprometido entra no build do MCP. Vetor mais difícil de eliminar: exige lockfiles, scan automático (Snyk/Trivy), revisão humana em atualizações.

Baixa probabilidade

Zero-day em infra base

Kernel, OpenSSL, runtime (Node/Python). Mitigação: updates automáticos da imagem, monitoramento de CVEs upstream, imagem distroless.

Média probabilidade

Insider ou credencial comprometida

Quem tem acesso legítimo pode abusar. Token roubado vira ação válida. Mitigação: princípio do menor privilégio + audit log + alertas em padrões anômalos.

Média probabilidade

Drift de configuração

Configurações migram com o tempo — TLS desabilitado para debug, IP whitelist relaxada, vault contornado. Mitigação: IaC versionado (Terraform) + auditoria trimestral.

Resumo honesto

Com domínio dedicado + hardening básico (mTLS/JWT + IP whitelist + vault + IaC + monitoramento), o MCP Server tem perfil de risco comparável a qualquer outro serviço corporativo bem mantido — não é "questão de tempo até invadirem". Os riscos reais que sobram são supply chain e drift de configuração, ambos mitigáveis com processo. Defensável para uso em produção.

Camadas obrigatórias para hospedagem segura

I
Rede
  • Atrás de Cloudflare ou WAF — nunca exposto direto na internet
  • IP whitelist: aceitar apenas conexões dos IPs publicados pela Anthropic
  • HTTPS obrigatório com certificado válido (Let's Encrypt auto-renovado)
  • Cloudflare Tunnel ou reverse proxy nginx com TLS termination
II
Autenticação
  • mTLS (cliente certificado) ou JWT assinado pelo SSO corporativo
  • Tokens por usuário com escopo limitado e expiração curta (15 min)
  • Sem autenticação básica HTTP — nunca
  • Login federado preferível (Okta, Azure AD)
III
Credenciais
  • HashiCorp Vault, AWS Secrets Manager ou similar — nunca .env em arquivo
  • Rotação automática (90 dias máximo)
  • Princípio do menor privilégio: cada API key com escopo mínimo
  • Quebra-vidros documentado para revogação emergencial
IV
Container e processo
  • Docker com user não-root
  • Imagem base mínima (distroless ou alpine)
  • Watchtower ou Renovate para updates automáticos da base
  • Scan da imagem (Trivy, Snyk) antes de cada deploy
V
Monitoramento
  • Logs estruturados (JSON) enviados ao SIEM corporativo
  • Alertas em pico de chamadas, IPs novos, erros 4xx/5xx em série
  • fail2ban para tentativas de brute force
  • Health check externo (UptimeRobot ou Pingdom)
VI
Continuidade
  • Backup criptografado das configurações (sem credenciais junto)
  • Plano de incidente documentado: quem para o serviço, quem revoga keys
  • Run book de recovery testado trimestralmente
  • Comunicação clara com analistas em caso de queda

Quando dá para fazer mais simples

Modo 1 · controle por firewall

MCP roda numa VPS comum (Hostinger, AWS, DigitalOcean) com firewall fechado para a internet aberta. Analistas alcançam por uma rede virtual entre máquinas — VPN corporativa existente ou Tailscale/Cloudflare Tunnel (tier gratuito). Pode usar IP direto, domínio é opcional. Caminho recomendado para Setter na maioria dos fluxos.

POC inicial

Variante temporária do Modo 1 para validar conceito: container Docker em VPS pequena + Cloudflare Tunnel (cria um endereço dedicado com proteção embutida) + token compartilhado entre poucos usuários resolve. Migrar para Modo 1 completo (ou Modo 2 se precisar de Code na Web) depois que provar valor.

Connectors públicos

Se o MCP Server só fala com APIs públicas (Wikipedia, Banco Central, dados públicos da B3), não há credenciais sensíveis a proteger. O foco passa a ser apenas disponibilidade.

Recomendação

Começar com Modo 1 (controle por firewall) como padrão. A VPS roda em qualquer provedor — Hostinger, AWS, DigitalOcean (~US$ 10/mês) — com firewall configurado para aceitar apenas IPs da rede da Setter (Tailscale free ou VPN corporativa existente). Esse MCP serve FactSet, S&P, Daloopa e o banco de consolidação para todo Code CLI/Desktop e Cowork rodando nas máquinas dos analistas. Pode usar IP direto, domínio é opcional. Apenas para fluxos com Code na Web — que roda na infra da Anthropic, fora da rede da Setter — adicionar Modo 2 (controle por autenticação): o mesmo MCP, mas com um endpoint específico em domínio dedicado (ex.: mcp.setter.com.br) protegido por Cloudflare Access (free até 50 usuários) com SSO + mTLS. Nunca deixar uma VPS sem firewall nem login — o problema não é a VPS, é deixá-la sem nenhuma das duas barreiras.

As duas formas reais de hospedar o MCP

Antes de comparar, três pontos que sempre confundem:

  • VPS = Virtual Private Server. O "Private" vem da virtualização (sua VPS é isolada de outros clientes do provedor) — nada a ver com acessibilidade na internet. Toda VPS contratada na Hostinger, AWS, DigitalOcean ou Azure vem com IP público por padrão — qualquer um na internet consegue tentar conectar nela. Sim, uma VPS Hostinger conta como VPS para os dois modos abaixo.
  • A diferença entre os dois modos não é onde o servidor está — em ambos é uma VPS de provedor de nuvem. A diferença é em que camada o acesso é controlado: no firewall (Modo 1) ou na autenticação da aplicação (Modo 2).
  • Domínio é opcional. O MCP responde no IP da VPS de qualquer jeito. Um domínio (mcp.setter.com.br) é só um nome amigável — ajuda em SSL e Cloudflare Access, mas não é obrigatório. No Modo 1 dá pra usar IP direto pela rede da Setter; no Modo 2, domínio facilita bastante mas tecnicamente IP nu também funciona.
Modo 1 · padrão recomendado

Acesso controlado pelo firewall (camada de rede)

Analogia: um telefone que só toca para números autorizados — quem não está na lista nem chega a fazer a chamada chegar.

Onde fica
VPS comum em provedor de nuvem (Hostinger, AWS, DigitalOcean). A VPS Hostinger funciona para esse modo — basta configurar o firewall.
Como o acesso é controlado
O firewall da VPS é configurado para recusar TUDO que vem da internet aberta. Aceita apenas conexões vindas de IPs específicos: os IPs dos analistas conectados na rede da Setter.
Como o analista chega
Conectando-se a uma rede virtual que a empresa controla. Pode ser:
  • A VPN corporativa que a Setter já usa para email/ERP, se existir;
  • Tailscale, Cloudflare Tunnel ou WireGuard — criam uma "rede entre máquinas" sobre a internet, com tier gratuito suficiente para uma gestora.
Ao conectar, o analista recebe um IP dessa rede (ex.: 100.64.5.3 no Tailscale). Esse IP entra na lista de autorizados do firewall.
Quem está fora dessa rede vê o quê?
Para qualquer atacante na internet aberta, o firewall responde "porta fechada" — nem chega a falar com o MCP. O servidor "não existe" para quem está fora.
Precisa de domínio?
Não obrigatoriamente. Analistas podem chamar o MCP pelo IP direto (ex.: https://100.64.5.3:8443). Domínio interno é opcional, só pra nome amigável.
Quando funciona
Code CLI, Code Desktop e Cowork rodando nas máquinas dos analistas — cobre >90% dos fluxos.
Quando NÃO funciona
Quando o Code na Web precisa chamar o MCP. Esse modo da Anthropic não consegue entrar na rede da Setter.
Modo 2 · necessário em casos específicos

Acesso controlado pela autenticação (camada de aplicação)

Analogia: um telefone público — qualquer um pode ligar — mas a secretária eletrônica pergunta a senha antes de passar a chamada adiante.

Onde fica
Mesmo tipo de VPS do Modo 1 (Hostinger, AWS, DigitalOcean). A configuração é que muda.
Como o acesso é controlado
O firewall aceita conexões de qualquer lugar. Mas a primeira coisa que o servidor MCP faz é exigir autenticação em três camadas: login SSO da Setter + certificado mTLS no cliente + IP autorizado. Sem isso, responde 401/403 e fecha — nenhum dado é entregue.
O que "alcançável da internet" significa aqui
Atacantes conseguem ver que o endereço existe (aparece em logs públicos de certificados SSL). Mas ver não é o mesmo que conseguir usar — passam horas tentando credenciais e tomam 401 em todas.
Precisa de domínio?
Praticamente sim. Cloudflare Access (que coloca o SSO na frente) requer domínio. Certificados SSL convencionais também. Tecnicamente IP nu funciona, mas a operação fica frágil — domínio é o caminho normal aqui.
Quando usar
Exclusivamente quando Code na Web precisa chamar o MCP. Esse modo roda em infra da Anthropic, fora da rede da Setter — então o servidor precisa estar alcançável pela internet, com a proteção em camadas descrita acima.
Custo adicional
Cloudflare Access (camada de SSO + mTLS): gratuito até 50 usuários, depois ~US$ 7/usuário/mês.
A terceira possibilidade (não usar) — para contraste

Tecnicamente, dá para alugar a mesma VPS (Hostinger, AWS, etc.) e deixá-la sem firewall e sem autenticação — porta aberta, qualquer um na internet entra sem barreira. Atacantes acham o IP em horas via Shodan, tentam credenciais comuns, e em algum momento conseguem entrar. Para uma gestora, não é opção real. O ponto importante: os Modos 1 e 2 acima também usam VPS (qualquer uma — Hostinger inclusive). O que muda é a configuração: o Modo 1 fecha o vetor na camada de rede (firewall), o Modo 2 fecha na camada de aplicação (login forte). O problema não é a VPS — é deixar a VPS sem nenhuma das duas barreiras.

Glossário · termos técnicos

Click para expandir. Padronize a linguagem do time.

VirtioFS

Protocolo de filesystem compartilhado entre VM e host. No Cowork, a pasta conectada do usuário é montada via VirtioFS dentro da VM. Arquivos escritos pelo agente aparecem instantaneamente no host — sem etapa de cópia.

MCP (Model Context Protocol)

Protocolo aberto da Anthropic para conectar Claude a sistemas externos (bancos, APIs, drives). Cada "connector" é um servidor MCP — pode ser stdio (local) ou remoto.

MCP Gateway

Proxy corporativo que centraliza credenciais. O agente do analista chama o gateway sem nunca ver a API key. Auditável, revogável, recomendado para qualquer adoção em escala.

Computer Use

Capacidade do Claude de controlar mouse, teclado e tela do usuário. No Cowork roda no host sem sandbox — é o vetor de prompt injection mais perigoso. Manter OFF por padrão.

Hooks (PreToolUse)

Pontos de interceptação que inspecionam e podem bloquear cada chamada de ferramenta antes da execução — enforcement em tempo real. Não confundir com audit log: hooks atuam no momento, mas não persistem trilha auditável centralizada. Cowork tem hooks mas não tem audit log; Code Enterprise tem ambos. Disponível em Cowork e Code, não em Chat.

Connectors

Integrações governadas com sistemas externos: FactSet, Google Drive, Slack, Daloopa, S&P. Implementadas via MCP. Diferente de "biblioteca instalada via pip" — connectors têm controle de permissão da Anthropic.

Sandbox

Isolamento kernel-level que restringe escrita ao diretório de trabalho e roteia rede via allowlist. No Code: Seatbelt (macOS), bubblewrap (Linux/WSL2). No Windows nativo: não suportado — usar Docker.

Managed Agents

Modo autônomo da plataforma Anthropic — Claude executa sessões longas com audit log nativo no Console. Defensável para compliance. Usado pelo modo Cookbook dos Financial Services.

Effort Control

Slider ao lado do seletor de modelo (Opus 4.8+). Aumentando, Claude raciocina mais profundamente (consumindo mais limite); reduzindo, responde mais rápido.
