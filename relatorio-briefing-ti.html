01 Enquadramento Contexto +

A Setter está implantando um servidor de integração de inteligência artificial para uso interno. O servidor será hospedado em uma VPS dedicada e consumido exclusivamente por aplicativos instalados nas máquinas dos colaboradores — sem exposição de dados públicos ou acesso de terceiros.

A equipe interna de IA & Dados gerencia a camada de aplicação (código, banco de dados, chaves de API). A prestadora de TI gerencia a infraestrutura (VPS, sistema operacional, segurança de rede e manutenção contínua do ambiente).

02 Dimensionamento Especificações da VPS +

2.1 Hardware

A aplicação é um processo Python com banco de dados SQLite — não é CPU-intensiva. O gargalo típico é latência de I/O de disco. Os valores abaixo refletem uso esperado com 5–15 usuários simultâneos e crescimento de dados de ~2–5 GB/ano.

2.3 DNS

Obrigatório — Um subdomínio será apontado para o IP da VPS. A Setter fornece o nome do subdomínio. O TI deve confirmar o IP estático para que o time interno crie o registro A, ou executar o apontamento se tiver acesso ao painel DNS da Setter.
03 Instalação Software a ser instalado pelo TI +

A camada de aplicação (código Python, banco de dados, containers) é gerenciada pela Setter e não é escopo do TI.

3.1 Pacotes do sistema Obrigatório

Todos os pacotes abaixo devem estar instalados antes da entrega:

bash · pacotes base
apt install -y \
  python3.11 python3.11-venv python3.11-dev python3-pip \
  git curl wget htop \
  nginx \
  certbot python3-certbot-nginx \
  fail2ban \
  ufw \
  sqlite3 \
  unattended-upgrades

3.2 nginx Obrigatório

Utilizado como reverse proxy com terminação TLS. O TI instala, habilita e confirma que o serviço está ativo. A configuração do server block (virtual host) é responsabilidade da Setter.

bash · habilitar nginx
systemctl enable nginx
systemctl start nginx

3.3 Certbot / Let's Encrypt Obrigatório

O TI instala o Certbot. A emissão do certificado é executada pela Setter após propagação do DNS.

3.4 Docker Opcional

A decisão de usar Docker para containerização é da Setter e será comunicada antes do deploy. O TI não é responsável por criar ou gerenciar containers — apenas por garantir que o ambiente suporte Docker caso seja necessário.

Ver instalação do Docker (Ubuntu 22.04)
bash · instalação oficial
# Instalação oficial para Ubuntu 22.04
curl -fsSL https://get.docker.com | sh
usermod -aG docker setter

3.5 WireGuard (VPN) Recomendado

Ver seção 5 completa sobre VPN. Se adotado, deve ser instalado pelo TI.

bash · wireguard
apt install -y wireguard

3.6 O que o TI não instala

  • Código da aplicação ou dependências Python específicas da aplicação
  • Banco de dados da aplicação
  • Containers Docker da aplicação
  • Chaves de API ou tokens de autenticação
04 Hardening Configuração de segurança +

4.1 Usuário dedicado da aplicação Obrigatório

Criar usuário sem privilégios de root para isolar a aplicação do sistema:

bash · usuário setter
adduser setter
usermod -aG sudo setter

O TI deve registrar a chave pública SSH da Setter neste usuário. O acesso SSH do próprio TI deve usar um usuário administrativo separado (ti-admin ou equivalente) — as credenciais das duas partes nunca se misturam.

4.2 Hardening de SSH

Aplicar em /etc/ssh/sshd_config. Reiniciar o serviço após as alterações. Validar login por chave SSH em sessão paralela antes de desabilitar autenticação por senha.

DiretivaValorClassificaçãoMotivo
PermitRootLoginnoObrigatórioElimina o vetor de ataque mais explorado em servidores Linux
PasswordAuthenticationnoObrigatórioForça uso exclusivo de chave SSH; elimina brute-force de senha
ChallengeResponseAuthenticationnoObrigatórioComplementa o item acima
MaxAuthTries3RecomendadoLimita tentativas por sessão SSH
ClientAliveInterval300RecomendadoDesconecta sessões inativas após 5 min
ClientAliveCountMax2RecomendadoComplementa o item acima
X11ForwardingnoRecomendadoDesabilita forwarding gráfico desnecessário em servidor

4.3 Firewall UFW Obrigatório

bash · regras ufw
ufw default deny incoming
ufw default allow outgoing
ufw allow 22/tcp    # SSH
ufw allow 80/tcp    # HTTP → redirect para HTTPS
ufw allow 443/tcp   # HTTPS (tráfego da aplicação)
ufw enable
Regra crítica: Nenhuma outra porta deve ser aberta. O processo da aplicação escuta exclusivamente em 127.0.0.1 (loopback) e não deve ser alcançável externamente de forma direta.

Se VPN for adotada (seção 5), substituir as regras 80/443 por:

bash · ufw com vpn
ufw allow 51820/udp  # WireGuard VPN
# Portas 80 e 443 podem ser removidas do acesso externo

4.4 Fail2ban Obrigatório

Configurar em /etc/fail2ban/jail.local:

Ver configuração jail.local + ativação
ini · /etc/fail2ban/jail.local
[DEFAULT]
bantime  = 3600
findtime = 600
maxretry = 5

[sshd]
enabled  = true
port     = ssh
logpath  = /var/log/auth.log
maxretry = 3

[nginx-http-auth]
enabled = true

[nginx-limit-req]
enabled = true
bash · ativar fail2ban
systemctl enable fail2ban
systemctl start fail2ban

4.5 Atualizações automáticas de segurança Obrigatório

bash · unattended-upgrades
apt install unattended-upgrades -y
dpkg-reconfigure --priority=low unattended-upgrades
# Selecionar "Yes"

Verificar que /etc/apt/apt.conf.d/20auto-upgrades contém:

conf · 20auto-upgrades
APT::Periodic::Update-Package-Lists "1";
APT::Periodic::Unattended-Upgrade "1";

4.6 Monitoramento de recursos Recomendado

Configurar alertas automáticos para o contato da Setter quando:

  • Uso de CPU > 80% por mais de 5 minutos
  • Uso de RAM > 85%
  • Uso de disco > 70%
  • Serviço nginx fora do ar

Ferramentas aceitas: qualquer sistema de monitoramento já utilizado pelo TI (Zabbix, Datadog, UptimeRobot, alerta simples por e-mail via cron). O importante é que o alerta chegue à Setter.

05 Rede privada VPN +

5.1 Por que VPN importa — classificação geral Recomendado

Sem VPN, o servidor fica acessível na internet pública via HTTPS — protegido por TLS e autenticação por token, mas exposto a varreduras, brute-force e DDoS. Com VPN, o servidor deixa de ser alcançável publicamente: somente dispositivos conectados ao túnel conseguem iniciar uma conexão.

Para o perfil da Setter — dispositivos corporativos — VPN representa ganho de segurança significativo com baixo custo operacional. Não é obrigatório para o deploy funcionar, mas é fortemente recomendado para ambiente de produção com dados sensíveis de M&A.

5.2 Topologia com VPN

diagrama · topologia
[Máquina do analista com cliente VPN]
        │  Túnel WireGuard criptografado
        ▼
[VPS — WireGuard Server]  ← único ponto de entrada exposto
        │
        └─ Rede interna do túnel → nginx → aplicação

Com esta topologia, 80/443 saem do UFW externo. Apenas a porta UDP 51820 fica exposta.

5.3 WireGuard Recomendado · solução preferencial

Nativo no kernel Linux desde 5.6 (Ubuntu 22.04 já inclui). Latência mínima, configuração auditável, sem dependência de terceiros.

Ver geração de chaves e arquivo wg0.conf
bash · chaves do servidor
apt install -y wireguard

# Gera par de chaves do servidor
wg genkey | tee /etc/wireguard/server_private.key | wg pubkey > /etc/wireguard/server_public.key
chmod 600 /etc/wireguard/server_private.key
ini · /etc/wireguard/wg0.conf
[Interface]
Address    = 10.10.0.1/24
PrivateKey = <SERVER_PRIVATE_KEY>
ListenPort = 51820

# Repetir bloco [Peer] para cada dispositivo autorizado
[Peer]
PublicKey  = <CHAVE_PÚBLICA_DO_PEER>
AllowedIPs = 10.10.0.2/32
bash · ativar wg0
systemctl enable wg-quick@wg0
systemctl start wg-quick@wg0

5.4 Tailscale Opcional · alternativa gerenciada

Baseado em WireGuard, elimina configuração manual de peers. Mais simples operacionalmente, mas cria dependência de serviço de terceiros para coordenação de acesso.

bash · tailscale
curl -fsSL https://tailscale.com/install.sh | sh
tailscale up

Usar apenas se o TI não tiver capacidade operacional para gerenciar WireGuard manualmente, e mediante aprovação da gestão da Setter quanto à dependência de terceiros.

5.5 Decisão a tomar na reunião

  • VPN será adotada? (RECOMENDADO, não obrigatório)
  • Se sim: WireGuard autogerenciado ou Tailscale?
  • Os dispositivos dos analistas são corporativos e controlados? (pré-requisito para VPN funcionar)

6.3 Snapshots da VPS Recomendado

Configurar snapshot automático semanal via painel do provedor cloud. Retenção mínima sugerida: 4 snapshots (1 mês). O TI é responsável por garantir que snapshots estejam ocorrendo — a Setter gerencia backup da camada de aplicação separadamente.

6.4 Reboots de kernel Obrigatório · procedimento

Quando um patch de kernel exigir reboot:

  1. Notificar a Setter com mínimo 24h de antecedência
  2. Aguardar confirmação da janela de manutenção
  3. Executar o reboot
  4. Confirmar que o servidor responde: curl https://[subdomínio]/health

A aplicação reinicia automaticamente via systemd — o TI não intervém na aplicação.

6.5 Fora do escopo do TI

O TI não é responsável por:

  • Restart ou debug da aplicação (Python / Uvicorn / Docker)
  • Alterações no banco de dados
  • Configuração ou rotação de API keys
  • Alterações no código ou dependências da aplicação
  • Configuração do server block do nginx

Qualquer incidente na camada de aplicação deve ser reportado à Setter para resolução interna.

07 Segregação de acesso Compartilhamento de credenciais e chaves +

7.1 Chaves SSH Obrigatório

  • A Setter fornece uma chave pública SSH para o usuário setter — acesso exclusivo da equipe interna
  • O TI mantém chave SSH própria em usuário administrativo separado (ti-admin ou equivalente)
  • Nenhuma senha de sistema é compartilhada entre as partes
  • Troca de chaves por canal seguro — Obrigatório (não por e-mail em texto plano)

7.2 API Keys da aplicação Obrigatório · restrição

As API keys são geradas e gerenciadas exclusivamente pela Setter. O TI não terá acesso a essas chaves e não deve solicitá-las. Estão armazenadas em arquivo com permissão 600, acessível apenas ao usuário setter.

7.3 Credenciais do console do provedor (out-of-band) Obrigatório

O acesso ao console de recuperação do provedor deve ser documentado e disponibilizado à Setter para uso em emergências onde SSH esteja inacessível. Definir na reunião o canal seguro para este compartilhamento — cofre de senhas corporativo (1Password, Bitwarden ou equivalente) é o método aceito.

7.4 Chaves WireGuard (se VPN adotada) Obrigatório (se VPN)

  • A chave pública do servidor WireGuard deve ser entregue à Setter para configuração dos peers
  • O TI armazena a chave privada do servidor com permissão 600
  • A Setter fornece as chaves públicas de cada dispositivo que precisa de acesso VPN
08 Aceite Checklist de entrega +

O TI deve validar e confirmar cada item abaixo antes de declarar a VPS pronta. A Setter não inicia o deploy sem este checklist completo.

Itens obrigatórios Obrigatório

  • VPS criada com Ubuntu 22.04 LTS e IP estático alocado
  • Usuário setter criado com chave SSH da Setter registrada
  • Login root desabilitado (PermitRootLogin no)
  • Autenticação por senha desabilitada (PasswordAuthentication no)
  • UFW ativo — somente portas 22, 80, 443 abertas (sudo ufw status verbose)
  • Fail2ban ativo (sudo fail2ban-client status)
  • unattended-upgrades configurado e ativo
  • nginx instalado e ativo (sem server block — não configurar virtual host)
  • Certbot instalado (certbot --version)
  • IP estático e credenciais SSH entregues à Setter por canal seguro
  • Acesso out-of-band documentado e procedimento de compartilhamento acordado

Itens recomendados Recomendado

  • Monitoramento de disco/CPU/RAM com alerta automático para a Setter
  • Snapshot automático semanal configurado no painel do provedor
  • Diretivas SSH adicionais aplicadas (MaxAuthTries, ClientAliveInterval, X11Forwarding)

Itens opcionais (se acordado) Opcional

  • Docker instalado e usuário setter no grupo docker
  • WireGuard instalado, serviço wg-quick@wg0 ativo, porta 51820/UDP aberta
  • Peer de teste VPN configurado e validado com a Setter
  • Procedimento de adição de novos peers VPN documentado
