## 1. Contexto

A Setter está implantando um servidor de integração de inteligência artificial para uso interno. O servidor será hospedado em uma VPS dedicada e consumido exclusivamente por aplicativos instalados nas máquinas dos colaboradores — sem exposição de dados públicos ou acesso de terceiros.

A aplicação roda **containerizada (Docker)**: um processo Python com banco de dados SQLite. A equipe interna de IA & Dados gerencia a **camada de aplicação** (código, imagens/containers, banco de dados, chaves de API e a lógica do script de backup). A prestadora de TI gerencia a **infraestrutura** (VPS, sistema operacional, Docker engine, rede, segurança, agendamento operacional de backup e manutenção contínua do ambiente).

> Os dados tratados envolvem operações de **M&A** — sensíveis. As decisões deste documento priorizam confidencialidade, disponibilidade e recuperabilidade.

## 2. Especificações da VPS

### 2.1 Hardware

A aplicação é um processo Python com banco de dados SQLite, executado em container — não é CPU-intensiva. O gargalo típico é latência de I/O de disco. Os valores abaixo refletem uso esperado com 5–15 usuários simultâneos e crescimento de dados de ~2–5 GB/ano.

| Recurso | Mínimo absoluto | Recomendado | Classificação | Observação |
|---|---|---|---|---|
| vCPU | 1 | 2 | `RECOMENDADO` | 1 core para aplicação + 1 para OS/nginx/Docker |
| RAM | 2 GB | 4 GB | `RECOMENDADO` | Aplicação usa ~200–400 MB; folga para picos e overhead do Docker |
| Disco | 20 GB SSD | 40 GB SSD | `RECOMENDADO` | SO + imagens Docker ~8 GB; banco + logs + backups locais crescem ~2–5 GB/ano |
| Tipo de disco | SSD | SSD NVMe | `RECOMENDADO` | SQLite é sensível a I/O; HDD causa lentidão visível |
| Rede | 100 Mbps | 200 Mbps | `RECOMENDADO` | Tráfego real baixo (~1–5 Mbps em pico) |
| IP estático | — | IPv4 dedicado | `OBRIGATÓRIO` | Necessário para WireGuard, DNS e certificado TLS |
| OS | — | Ubuntu 24.04 LTS | `OBRIGATÓRIO` | Decisão v2. Não serão aceitas outras distros |

> **Mudança v2 — base do SO:** a base é **Ubuntu 24.04 LTS** (não mais 22.04). A aplicação roda em container, então a versão do Python é definida pela imagem Docker da Setter, não pelo SO do host. O 24.04 entrega toolchain atual e suporte de longo prazo (até 2029).

## 3. Software a Ser Instalado pelo TI

> **Decisão v2 — arquitetura em Docker:** a aplicação roda containerizada. O TI prepara o **host** (Docker engine, nginx, segurança), mas **não instala Python nem dependências da aplicação no SO** — isso vai dentro da imagem Docker, gerenciada pela Setter. O TI nunca toca no conteúdo dos containers.

### 3.1 Pacotes do sistema (host) — `OBRIGATÓRIO`

Todos os pacotes abaixo devem estar instalados antes da entrega. Observe que **não há Python da aplicação aqui**:

```bash
apt update && apt install -y \
  git curl wget htop \
  nginx \
  certbot python3-certbot-nginx \
  fail2ban \
  ufw \
  sqlite3 \
  logrotate \
  unattended-upgrades \
  wireguard
```

> `sqlite3` permanece no host apenas como ferramenta para o script de backup operar sobre o arquivo do banco (ver 6.3). `certbot python3-certbot-nginx` pode exigir um plugin de DNS-01 conforme o provedor de DNS da Setter (ver 5.4) — alinhar na reunião.

### 3.2 Docker — `OBRIGATÓRIO`

A aplicação roda em container. O TI instala e habilita o Docker engine, mas **não cria nem gerencia os containers da aplicação** — isso é responsabilidade da Setter.

```bash
curl -fsSL https://get.docker.com | sh
systemctl enable docker
systemctl start docker
```

> **Privilégio:** pertencer ao grupo `docker` equivale a root. A conta de serviço `setter-app` (ver 4.1) **não** entra no grupo `docker`. Apenas contas administrativas humanas (`setter-admin` / `ti-admin`) gerenciam o Docker, e os containers da aplicação rodam como usuário **não-root** (via `--user` ou user-namespace remap).

### 3.3 nginx — `OBRIGATÓRIO`

Reverse proxy com terminação TLS. O TI instala, habilita e confirma que o serviço está ativo. A configuração do `server block` (virtual host) é responsabilidade da Setter, mas deve respeitar os mínimos de TLS e rate limiting da seção 4.9.

```bash
systemctl enable nginx
systemctl start nginx
```

### 3.4 Certbot / Let's Encrypt — `OBRIGATÓRIO`

O TI instala o Certbot. Como a VPS fica atrás de VPN e **não expõe 80/443 publicamente** (ver 4.3), a emissão e a renovação do certificado usam o desafio **DNS-01**, não HTTP-01. A emissão inicial é executada pela Setter; a renovação automática entra no escopo contínuo do TI (ver 6.1).

### 3.5 WireGuard (VPN) — `OBRIGATÓRIO`

A VPN foi adotada como **caminho único** (decisão v2 — ver seção 5). O pacote já está no bloco 3.1. A configuração completa do túnel é detalhada na seção 5.

### 3.6 O que o TI não instala

- Python da aplicação, bibliotecas ou dependências (vão na imagem Docker da Setter)
- Conteúdo, imagens ou containers da aplicação
- Banco de dados da aplicação
- Chaves de API ou tokens de autenticação
- Lógica do script de backup (a Setter escreve; o TI apenas agenda — ver 6.3)

## 4. Configuração de Segurança

### 4.1 Contas e segregação de privilégio — `OBRIGATÓRIO`

Para honrar o modelo de menor privilégio, separam-se a **conta de serviço** (dona dos arquivos da aplicação, sem poder administrativo) das **contas humanas administrativas**:

| Conta | sudo? | grupo docker? | Função |
|---|---|---|---|
| `setter-app` | **NÃO** | **NÃO** | Conta de serviço. Dona dos arquivos da aplicação, volumes e do arquivo do banco SQLite. Sem login interativo. |
| `setter-admin` | Sim | Sim | Administrador humano da Setter. Faz deploy e gerencia containers. Chave SSH da Setter. |
| `ti-admin` | Sim | Sim | Administrador humano do TI. Administra a infraestrutura. Chave SSH do TI. |

```bash
# Conta de serviço — sem sudo, sem shell de login, sem docker
adduser --disabled-password --gecos "" setter-app

# Administradores humanos — com sudo
adduser setter-admin && usermod -aG sudo,docker setter-admin
adduser ti-admin     && usermod -aG sudo,docker ti-admin
```

> **Por que separar:** se a aplicação for comprometida (ex.: prompt injection no MCP), o atacante herda apenas os poderes de `setter-app` — que não tem `sudo` nem `docker`. O processo nunca roda com poder de root. A chave pública SSH da Setter é registrada em `setter-admin`; a do TI em `ti-admin`. As credenciais das duas partes nunca se misturam.

### 4.2 Hardening de SSH

Aplicar em `/etc/ssh/sshd_config`. Reiniciar o serviço após as alterações. **Validar login por chave SSH em sessão paralela antes de desabilitar autenticação por senha.**

| Diretiva | Valor | Classificação | Motivo |
|---|---|---|---|
| `PermitRootLogin` | `no` | `OBRIGATÓRIO` | Elimina o vetor de ataque mais explorado em servidores Linux |
| `PasswordAuthentication` | `no` | `OBRIGATÓRIO` | Força uso exclusivo de chave SSH; elimina brute-force de senha |
| `ChallengeResponseAuthentication` | `no` | `OBRIGATÓRIO` | Complementa o item acima |
| `MaxAuthTries` | `3` | `RECOMENDADO` | Limita tentativas por sessão SSH |
| `ClientAliveInterval` | `300` | `RECOMENDADO` | Desconecta sessões inativas após 5 min |
| `ClientAliveCountMax` | `2` | `RECOMENDADO` | Complementa o item acima |
| `X11Forwarding` | `no` | `RECOMENDADO` | Desabilita forwarding gráfico desnecessário em servidor |

### 4.3 Firewall UFW — `OBRIGATÓRIO`

Com a VPN adotada como caminho único, o tráfego da aplicação passa pelo túnel. O nginx escuta na interface do túnel; **80/443 não são abertas ao público**.

```bash
ufw default deny incoming
ufw default allow outgoing
ufw allow 22/tcp     # SSH (admin) — protegido por chave + fail2ban
ufw allow 51820/udp  # WireGuard (único ponto de entrada público)
ufw enable
# 80/443 NÃO são expostas à internet pública.
# nginx escuta na interface do túnel (ex.: 10.10.0.1); a aplicação em 127.0.0.1.
```

> **Regra crítica:** nenhuma outra porta deve ser aberta ao público. O processo da aplicação escuta exclusivamente em `127.0.0.1` (loopback) dentro do host e só é alcançável via túnel WireGuard.

### 4.4 Fail2ban — `OBRIGATÓRIO`

Configurar em `/etc/fail2ban/jail.local`:

```ini
[DEFAULT]
bantime  = 3600
findtime = 600
maxretry = 5

[sshd]
enabled  = true
port     = ssh
logpath  = /var/log/auth.log
maxretry = 3
```

```bash
systemctl enable fail2ban
systemctl start fail2ban
```

> As jails `nginx-*` da v1 foram removidas do conjunto base porque, no modo VPN, o nginx não recebe tráfego público direto. Se a Setter expuser algum endpoint, reavaliar.

### 4.5 Atualizações automáticas de segurança — `OBRIGATÓRIO`

```bash
apt install unattended-upgrades -y
dpkg-reconfigure --priority=low unattended-upgrades
# Selecionar "Yes"
```

Verificar que `/etc/apt/apt.conf.d/20auto-upgrades` contém:

```
APT::Periodic::Update-Package-Lists "1";
APT::Periodic::Unattended-Upgrade "1";
```

### 4.6 Monitoramento de recursos — `RECOMENDADO`

Configurar alertas automáticos para o contato da Setter quando:

- Uso de CPU > 80% por mais de 5 minutos
- Uso de RAM > 85%
- Uso de disco > 70%
- Serviço nginx, docker ou wg-quick@wg0 fora do ar

Ferramentas aceitas: qualquer sistema de monitoramento já utilizado pelo TI (Zabbix, Datadog, UptimeRobot, alerta simples por e-mail via cron). O importante é que o alerta chegue à Setter (ver canal na seção 9).

### 4.7 Monitoramento da saúde da aplicação — `OBRIGATÓRIO`

> Monitorar recursos (4.6) não basta: a aplicação pode estar "no ar" e devolvendo erro. É obrigatório um monitor que confirme que a aplicação **responde**.

Como a aplicação fica atrás da VPN, o monitor deve alcançar o endpoint **de dentro do túnel** ou rodar no próprio host:

```bash
# a cada 5 min: bate no /health e alerta a Setter se falhar 2x seguidas
*/5 * * * * curl -fsS --max-time 10 http://127.0.0.1:PORTA/health \
  || /usr/local/bin/alerta-setter.sh "Aplicação não respondeu ao /health"
```

Alternativa: um monitor externo (UptimeRobot ou equivalente) configurado como **peer do túnel**, batendo no endpoint via IP do WireGuard. A Setter define o caminho exato do `/health`; o TI implementa o monitor e o alerta.

### 4.8 Rotação de logs (logrotate) — `OBRIGATÓRIO`

Log que não rotaciona enche o disco e derruba o serviço. A Setter informa o caminho dos logs da aplicação; o TI configura o `logrotate` uma vez.

```
# /etc/logrotate.d/aplicacao-setter
/caminho/dos/logs/aplicacao/*.log {
    daily
    rotate 14
    compress
    delaycompress
    missingok
    notifempty
    copytruncate
}
```

### 4.9 Mínimos de TLS e rate limiting — `OBRIGATÓRIO`

O `server block` é configurado pela Setter, mas este documento fixa os mínimos de segurança que ele deve respeitar — dados de M&A:

- **TLS 1.2 ou superior**; protocolos antigos (SSLv3, TLS 1.0/1.1) desabilitados (`ssl_protocols TLSv1.2 TLSv1.3;`).
- Cifras modernas; HSTS habilitado.
- **Rate limiting no endpoint do MCP/aplicação** (não apenas no auth) via `limit_req` do nginx — consistente com o modelo de ameaça (prompt injection / abuso de endpoint).

```nginx
limit_req_zone $binary_remote_addr zone=mcp:10m rate=10r/s;

location /mcp/ {
    limit_req zone=mcp burst=20 nodelay;
    proxy_pass http://127.0.0.1:PORTA;
}
```

## 5. VPN — WireGuard (caminho único)

> **Decisão fechada (v2):** a VPN foi adotada via **WireGuard autogerenciado**. Este é o caminho único entregue ao TI — o documento não descreve mais o cenário "sem VPN". O servidor deixa de ser alcançável publicamente: somente dispositivos no túnel iniciam conexão.

### 5.1 Por que WireGuard

Sem VPN, o servidor ficaria exposto a varreduras, brute-force e DDoS. Com WireGuard, o único ponto de entrada público é a porta UDP 51820. Nativo no kernel Linux (Ubuntu 24.04 já inclui), latência mínima, configuração auditável, sem dependência de terceiros — adequado a dados sensíveis de M&A.

### 5.2 Topologia

```
[Máquina do analista com cliente WireGuard]
        │  Túnel WireGuard criptografado
        ▼
[VPS — WireGuard Server]  ← único ponto de entrada exposto (51820/UDP)
        │
        └─ Rede interna do túnel → nginx → aplicação (container, 127.0.0.1)
```

### 5.3 Configuração do servidor — `OBRIGATÓRIO`

```bash
# Gera par de chaves do servidor
wg genkey | tee /etc/wireguard/server_private.key | wg pubkey > /etc/wireguard/server_public.key
chmod 600 /etc/wireguard/server_private.key
```

Arquivo `/etc/wireguard/wg0.conf`:

```ini
[Interface]
Address    = 10.10.0.1/24
PrivateKey = <SERVER_PRIVATE_KEY>
ListenPort = 51820

# Repetir bloco [Peer] para cada dispositivo autorizado
[Peer]
PublicKey  = <CHAVE_PÚBLICA_DO_PEER>
AllowedIPs = 10.10.0.2/32
```

```bash
systemctl enable wg-quick@wg0
systemctl start wg-quick@wg0
```

### 5.4 Impacto no certificado TLS

> Como 80/443 não ficam públicas, o desafio **HTTP-01 do Let's Encrypt não funciona**. A emissão e a renovação do certificado devem usar **DNS-01** (não exige porta pública). Isso pode exigir um plugin de DNS do certbot conforme o provedor de DNS da Setter — alinhar na reunião quem detém o token de API do DNS e onde ele é armazenado.

### 5.5 Validação de saúde por cenário

O health check da v1 (`curl https://[subdomínio]/health` de fora) não funciona atrás da VPN. Validação definida:

- **No host:** `curl -fsS http://127.0.0.1:PORTA/health` (usado pelo cron de 4.7 e pós-reboot).
- **De fora (analista/monitor):** conectar ao túnel e bater no IP interno — `curl -fsS https://10.10.0.1/health` ou no subdomínio que resolve para o IP do túnel.

### 5.6 Adição de peers

A Setter fornece a chave pública de cada dispositivo. O TI adiciona o bloco `[Peer]` e documenta o procedimento no as-built (ver seção 10). Tailscale foi avaliado e **não adotado** (evita dependência de terceiros).

## 6. Manutenção Contínua e Backup — Escopo do TI

### 6.1 Processos automáticos — `OBRIGATÓRIO` (configurar uma vez)

|---|---|---|
| Patches de segurança do OS | `unattended-upgrades` | Aplicação automática diária |

| Banimento de IPs suspeitos | Fail2ban | Proteção contínua sem intervenção |
| Backup do banco de dados | Script da Setter via cron (ver 6.3) | Backup diário enviado para fora da VPS |
| Rotação de logs | `logrotate` | Logs comprimidos e expurgados; disco não enche |
| Health check da aplicação | cron + curl (ver 4.7) | Alerta à Setter se o `/health` falhar |

### 6.2 Revisão periódica — `OBRIGATÓRIO`

| Tarefa | Frequência |
|---|---|
| Revisão de `/var/log/auth.log` — tentativas de acesso suspeitas | Mensal |
| Verificação de IPs banidos pelo fail2ban | Mensal |
| Verificação de espaço em disco — alertar Setter se > 70% | Mensal |
| Confirmação de que os backups estão chegando ao destino externo | Mensal |
| Teste de restauração do backup em ambiente limpo (ver 6.3) | Trimestral |
| Verificação de pacotes fora do escopo do `unattended-upgrades` | Mensal |

### 6.3 Backup do banco de dados — `OBRIGATÓRIO`

> **Lacuna corrigida na v2.** Na v1 o backup do banco ficou "fora do escopo do TI" e sem dono operacional — justamente o ativo mais crítico. Agora a responsabilidade é dividida e explícita.

| Responsável | O quê |
|---|---|
| **Setter** | Escreve o script de backup (a lógica é da aplicação): usa `sqlite3 .backup` ou `.dump` — nunca cópia do arquivo aberto. |
| **TI** | Agenda o script via `cron`, garante o **envio da saída para fora da VPS** (object storage / bucket externo), monitora falhas e executa o teste de restauração. |

> **Por que não basta snapshot:** snapshot é imagem de disco inteira, ponto-no-tempo grosseiro — não dá recuperação granular de uma tabela corrompida ou de um registro apagado por engano. E **copiar um arquivo SQLite aberto corrompe** — por isso o backup precisa usar `.backup`/`.dump` do próprio SQLite.

```bash
# 02h todo dia: executa o script da Setter e envia para destino externo
0 2 * * * /opt/setter/backup-db.sh && /opt/setter/enviar-backup-externo.sh
```

**Teste de restauração — `OBRIGATÓRIO`:** backup que ninguém restaurou não é backup. O TI restaura o backup em ambiente limpo e confirma a integridade — uma vez na entrega e depois trimestralmente.

**RPO / RTO — metas definidas:**

| Métrica | Meta | Significado |
|---|---|---|
| **RPO** (perda máxima aceitável) | ≤ 24h | Backup diário. Pode ser reduzido para 6h se a Setter exigir. |
| **RTO** (tempo para voltar) | ≤ 4h | Da detecção à aplicação restaurada a partir do último backup íntegro. |

> Valores propostos; confirmar/ajustar na reunião. Sem RPO/RTO, "poucas preocupações" não é mensurável.

### 6.4 Snapshots da VPS — `OBRIGATÓRIO`

**Responsabilidade do TI:** configurar e garantir o snapshot automático semanal via painel do provedor. Retenção mínima: 4 snapshots (1 mês). **O snapshot complementa, mas não substitui o backup do banco (6.3)** — serve para recuperação de desastre do host inteiro, não para recuperação granular de dados.

> **Atenção de segurança:** o snapshot captura o disco inteiro, incluindo o arquivo de chaves da aplicação. Quem controla snapshots controla os segredos — ver tratamento na seção 7.2.

### 6.5 Reboots de kernel — `OBRIGATÓRIO` (procedimento)

Quando um patch de kernel exigir reboot:

1. Notificar a Setter com mínimo 24h de antecedência
2. Aguardar confirmação da janela de manutenção
3. Executar o reboot
4. Confirmar que a aplicação responde no host: `curl -fsS http://127.0.0.1:PORTA/health` e que `wg-quick@wg0` subiu

> **Resiliência da aplicação (corrigido na v2):** a afirmação "a aplicação reinicia sozinha" só é verdadeira se a unit systemd / política do container tiver `Restart=always` + `StartLimitIntervalSec` (proteção contra crash-loop). A configuração da unit é da Setter; a **validação** (matar o processo e confirmar que volta sozinho) entra no checklist de entrega — ver seção 8.

### 6.6 Fora do escopo do TI

O TI **não é responsável** por:

- Restart ou debug da aplicação (Python / Uvicorn / container)
- Alterações no banco de dados ou na lógica do script de backup
- Configuração ou rotação de API keys
- Alterações no código, imagens ou dependências da aplicação
- Configuração do `server block` do nginx

Qualquer incidente na camada de aplicação deve ser reportado à Setter para resolução interna.

## 7. Compartilhamento de Credenciais e Chaves

### 7.1 Chaves SSH e offboarding — `OBRIGATÓRIO`

- A Setter fornece uma chave pública SSH para o usuário `setter-admin` — acesso exclusivo da equipe interna
- O TI mantém chave SSH própria em `ti-admin`
- Nenhuma senha de sistema é compartilhada entre as partes
- Troca de chaves por canal seguro — `OBRIGATÓRIO` (não por e-mail em texto plano)

> **Política de offboarding (novo na v2):** quando o prestador trocar de funcionário ou o contrato encerrar, o TI **notifica a Setter e revoga as chaves SSH em até 24h**. A Setter mantém um inventário das chaves autorizadas e audita o `authorized_keys` de cada conta na revisão mensal. O gatilho de revogação não pode depender de memória de uma pessoa.

### 7.2 API Keys da aplicação — `OBRIGATÓRIO` (restrição)

As API keys são geradas e gerenciadas exclusivamente pela Setter. O TI **não terá acesso direto** a essas chaves e **não deve solicitá-las**. Estão em arquivo com permissão 600, acessível apenas a `setter-app`.

> **Contradição reconhecida e resolvida (v2):** a permissão 600 protege as chaves no sistema em execução, mas **quem controla snapshots ou o console do provedor consegue ler o disco inteiro** — incluindo o arquivo de chaves. Como o TI é responsável pelos snapshots (6.4) e detém o console out-of-band (7.3), na prática a conta do provedor é uma superfície que carrega segredos. Por isso ela é tratada com o mesmo rigor das chaves: **MFA obrigatório** (2.2), credenciais em cofre (7.3) e acesso restrito ao mínimo de pessoas. A Setter aceita este modelo em vez de adotar um secrets manager externo, dado o porte da operação.

### 7.3 Credenciais do console do provedor (out-of-band) — `OBRIGATÓRIO`

O acesso ao console de recuperação deve ser documentado e disponibilizado à Setter para emergências onde o SSH esteja inacessível. **MFA obrigatório** na conta. As credenciais ficam em cofre de senhas corporativo (1Password, Bitwarden ou equivalente) — nunca em e-mail ou texto plano. Definir na reunião o cofre e quem tem acesso.

### 7.4 Chaves WireGuard — `OBRIGATÓRIO`

- A chave pública do servidor WireGuard é entregue à Setter para configuração dos peers
- O TI armazena a chave privada do servidor com permissão 600
- A Setter fornece as chaves públicas de cada dispositivo que precisa de acesso VPN

### 7.5 Notificação de incidente de segurança — `OBRIGATÓRIO`

Se o TI detectar indício de comprometimento (acesso não autorizado, escalonamento de privilégio, exfiltração), deve notificar a Setter conforme o SLA crítico da seção 9 — **em até 1h, por canal direto** — preservando logs e evidências. Não tomar ação destrutiva (reinstalar, apagar) sem alinhamento, exceto isolar a VPS da rede se houver exfiltração ativa.

## 9. SLA, Contato e Escalação

### 9.1 SLA de atendimento por severidade — `OBRIGATÓRIO`

> Sem tempos de resposta acordados, a disponibilidade fica refém da boa vontade do prestador. A tabela abaixo define a obrigação de agir por severidade.

| Severidade | Exemplos | Tempo de resposta | Cobertura |
|---|---|---|---|
| **Crítico** | Serviço fora do ar, VPS inacessível, suspeita de comprometimento de segurança | **1 hora** | 24/7 |
| **Alto** | Certificado expira em < 72h, backup falhando, túnel WireGuard instável | 4 horas úteis | Horário comercial |
| **Médio** | Disco > 70%, tentativas de acesso suspeitas | 1 dia útil | Horário comercial |
| **Baixo** | Dúvidas, ajustes não urgentes, melhorias | 3 dias úteis | Horário comercial |

> **Cobertura fora do horário:** apenas severidade **Crítica** é atendida 24/7 (1h). As demais são tratadas no próximo horário comercial (seg–sex, 9h–18h). Valores propostos — confirmar contratualmente na reunião. Mesmo que a resposta seja "só horário comercial", o risco fica conhecido.

### 9.2 Contato e bus factor — `OBRIGATÓRIO`

> Alerta que vai para uma só pessoa morre se ela estiver de férias ou doente (bus factor de 1). Os alertas devem ir para um **canal/lista compartilhada**, não para um e-mail pessoal único.

| Papel | Contato |
|---|---|
| Canal primário de alertas (lista/grupo) | *[definir — lista ou canal de equipe, ex.: ia-dados@setter.net.br]* |
| Contato primário | Thiago Bagdonas \| thiago.bagdonas@setter.net.br |
| Contato secundário (backup) | *[definir na reunião]* |

### 9.3 Matriz de escalação

| Situação | Ação esperada do TI |
|---|---|
| VPS inacessível via SSH | Acionar console out-of-band e notificar Setter imediatamente (Crítico — 1h) |
| Suspeita de comprometimento | Notificar em até 1h, preservar evidências, isolar se houver exfiltração (ver 7.5) |
| Aplicação não responde ao /health | Confirmar infra (nginx/docker/túnel) e notificar Setter; não debugar a aplicação |
| Backup falhando | Investigar agendamento/envio externo e notificar Setter (Alto) |
| Alerta de disco > 70% | Notificar Setter para providências na camada de aplicação |
| Certificado TLS perto de expirar | Verificar renovação DNS-01 (`certbot renew`) e confirmar para a Setter |
| Patch de kernel disponível | Notificar Setter, agendar reboot em janela acordada (ver 6.5) |
| Qualquer dúvida sobre a aplicação | Reportar à Setter — não intervir na aplicação sem autorização |

## 10. Runbook de Recuperação e Documentação As-Built

### 10.1 Documento as-built — `OBRIGATÓRIO`

Na entrega, o TI fornece um documento "as-built" registrando o **estado real do servidor**, atualizado a cada mudança relevante. Sem isso, daqui a seis meses ninguém sabe o que foi configurado. Deve conter:

- Provedor, plano, região e IP estático
- Versões instaladas (SO, Docker, nginx, certbot, WireGuard)
- Caminhos relevantes (logs, volumes, arquivo do banco, scripts de backup, wg0.conf)
- Faixa de IP do túnel e peers configurados
- Onde está o destino externo do backup e como é acessado
- Decisões tomadas e quaisquer desvios deste briefing (justificados por escrito)

### 10.2 Runbook de recuperação — `RECOMENDADO`

Procedimentos curtos para os cenários mais prováveis. Reduzem a dependência de uma pessoa específica estar disponível.

| Cenário | Quem age | Passos resumidos |
|---|---|---|
| **Aplicação não sobe** | Setter (TI confirma infra) | TI confirma docker/nginx/túnel ativos e logs de sistema; Setter inspeciona o container e logs da aplicação; validar `Restart=always` |
| **Disco cheio** | TI | Identificar maior consumidor (logs, imagens órfãs, backups locais); aplicar `logrotate`/`docker system prune`; alertar Setter; reavaliar retenção |
| **Renovação de certificado falhou** | TI | Verificar token de DNS-01 e logs do certbot; rodar `certbot renew` manualmente; confirmar com a Setter |
| **Banco corrompido** | Setter (TI fornece backup) | Parar a aplicação; TI disponibiliza o último backup íntegro do destino externo; Setter restaura via `.restore`/`.dump`; validar integridade; respeitar RTO ≤ 4h |
